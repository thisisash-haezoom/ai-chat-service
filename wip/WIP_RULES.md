# 작업 일지 규칙 (WIP Rules)

## 📋 목적
모든 개발 작업의 진행 상황, 트러블슈팅, 기능 업데이트를 체계적으로 기록

## 📁 폴더 구조
```
/wip
├── WIP_RULES.md          (이 파일)
├── 2026-02-27.md         (날짜별 작업 일지)
├── 2026-02-28.md
└── ...
```

## 📝 파일 작성 규칙

### 파일명
- 형식: `YYYY-MM-DD.md`
- 예: `2026-02-27.md`

### 파일 구조
```markdown
# 📅 2026-02-27 작업 일지

## ✅ 완료 작업

### 1. [작업 제목]
- **파일**: `src/path/to/file.tsx`
- **변경 내용**: 무엇을 했는지 간단히
- **관련 기능**: 어떤 기능에 영향을 미쳤는지

### 2. [다음 작업...]

## 🔧 트러블슈팅

### 1. [문제점]
- **원인**: 무엇이 문제였는가
- **해결책**: 어떻게 해결했는가
- **학습점**: 다음에 피할 점

### 2. [다음 문제...]

## 🚀 업데이트된 기능

### 1. [기능명]
- **설명**: 새로 추가되거나 개선된 기능
- **사용 방법**: 어떻게 사용하는가
- **상태**: ✅ 완료 / ⚠️ 진행 중 / ❌ 미완료

### 2. [다음 기능...]

## 📌 주요 통계
- **총 작업 수**: X개
- **완료된 작업**: X개
- **트러블슈팅**: X건
- **새로운 기능**: X개

## 🔗 참고 링크
- 관련 코드: 파일 경로들
- 관련 API: API 엔드포인트들
```

## 작성 규칙

### ✅ 완료 작업
- 파일 경로를 명시적으로 기재
- 변경 전후를 간단히 설명
- 한 작업 = 한 섹션

### 🔧 트러블슈팅
- 마주친 모든 에러/문제를 기록
- 원인과 해결책을 명확히
- 다른 개발자가 참고할 수 있도록 작성

### 🚀 업데이트된 기능
- 사용자 입장에서 변화를 설명
- 기능이 어떻게 동작하는지 설명
- 필요하면 예시 코드 추가

## 💡 팁
1. **매일 끝날 때 기록하기** - 작은 것도 빠뜨리지 말기
2. **명확한 언어** - 다른 개발자도 이해할 수 있도록
3. **코드 스니펫** - 복잡한 변경은 코드 예시 포함
4. **관련 파일 링크** - 찾기 쉽도록 경로 명시
5. **시간 예상** - 다음 작업 계획에 도움이 됨

## 🔄 검토 체크리스트
- [ ] 모든 작업이 명확히 설명되었는가?
- [ ] 트러블슈팅 해결책이 재현 가능한가?
- [ ] 새 기능이 어떻게 동작하는지 명확한가?
- [ ] 파일 경로가 정확한가?
- [ ] 통계가 정확한가?
